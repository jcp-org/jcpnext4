Minutes

We reviewed the issues entered from discussion last week and also reviewed the JSR 358 issues opened to determine which issues could be pulled into this JSR. We continued our review the Process Document, and marked it up where changes will be needed. See marked up Process Document.

Issues to be moved over (or apply to both JSRs/move if we can do so): 27, 38, 43, 46, 13, 4, 6, 10, 11, 19, 28, 50, 51, 52, 12.

Issues to enter for inclusion in this JSR:

Section 1.1.2 - add 'must'

Section 2.4 Form the Expert Group. Fix grammar and add process for forming contributors for the JSR, open to Affiliate Level Members.

Section 1.1 Insert reference to JSR Review Process and the publication of various stats, including those relating to Transparency, linked from the JSR page on jcp.org.



Issues to enter for possible inclusion in this JSR:


Add Incubator stage in Process Document. Should we introduce some kind of “stage 0” (incubator) process – or make this part of the initiation process?


Section 1.1 Insert requirement for collaborative (“open source”) development process for RI. Provide a pointer to the collaboration project’s “home page”. We should define at a high level the characteristics we require in the collaboration project.


Section 1.3 JSR Deadlines. Should we shorten the time-periods in the interests of more rapid posting of milestones?


Section 2.1.1 and Section 5.1.1. Can we add anything here to make the transition of a JSR from one Spec Lead to another easier (and from ML to another)? Should we insert a note “encouraging” the ML to transfer IP to the new lead (without this it will effectively be impossible for someone else to take over)? Depends on how the output of the JSR is licensed.


Section 3.2, 3.3 and 3.4. Should the review period be extended when JSRs publish updates to their milestone drafts? PMO to do this automatically when this happens? Should we prohibit changes during review (to avoid the confusion of people reviewing different versions) and instead insist on a second draft later? Especially important for 3.3 and 3.4 when there is an EC Ballot.


Section 4.3. Require a statement from Spec Lead on expectations with regard to future (follow-on) work. Do they expect to issue Maintenance Releases, or even a new JSR (version x+1)?


Section 5. Should we tighten up the distinction between what is appropriate for a MR and what should go into a new JSR?


Section 5.2. An issue that was referenced in a "conditional yes" vote (explain the term? Yes with comment such as “I’m not voting no now, but I expect issue X to be addressed in the future) during an earlier development stage has not been addressed.


Appendix A. There is no Reference Implementation or Technology Compatibility Kit to be delivered and no TCK appeals process to be defined. Should we define some kind of testing process to verify whether we are meeting the requirements of the Process Document.




Next Steps:

Consult Spec Leads for their ideas on possible changes to the Process Document.

Submit JSR for posting on JCP.org once approvals received.

Present progress next Tuesday at the JCP EC Teleconference at 07:00 PDT.
